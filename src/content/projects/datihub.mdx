## El problema

DatiHub necesitaba un backend que procesara conversaciones a gran escala. La base de código original tenía funciones de cientos de líneas sin separación clara entre lógica de negocio e infraestructura.

## Mi aporte

Lideré la reestructuración del backend aplicando Clean Architecture con CQRS para separar commands de queries. Esto permitió escalar lecturas y writes de forma independiente según el tráfico real.

### Arquitectura de flujos

Diseñé un sistema de flujos con Builder Pattern para crear conversaciones condicionales sin hardcodear lógica — mensajes, condiciones basadas en contexto, auto-advance con delays configurables.

### Refactor de la base de código

- Separación de lógica de negocio de infraestructura
- Implementación de **tsyringe** como DI container para manejar dependencias de forma limpia
- Repositorio abstracto + datasource para desacoplar el ORM
- Sistema de errores jerárquico con distinción entre errores operacionales y de programa

### Infraestructura

- PostgreSQL + Prisma para persistencia
- Redis para caché y pub/sub
- PGMQ como cola de mensajes persistente (recuperación ante fallos)
- Implementando observabilidad con Grafana + Prometheus
- Stress testing y optimización de queries para concurrencia

## Decisiones técnicas

**¿Por qué PGMQ sobre Redis Streams?**
La cola persistente era importante para recuperación ante fallos. PGMQ sobre Postgres dio persistencia + operaciones atómicas sin mantener otro sistema.

**CQRS desde la reestructuración**
El volumen de lecturas es 10x mayor que writes. CQRS permitió optimizar queries sin complicar el modelo de dominio.

**tsyringe como DI container**
Pasamos de inyección manual inconsistente a un contenedor liviano que simplificó las dependencias sin agregar magia.
